# Harn-Gan vs KBank Kuntong — ข้อดีข้อด้อยของทั้งคู่

## สรุปสั้น
ถ้ามองจาก “ปัญหาที่แก้” ทั้งสองตัวอยู่ในโลกเดียวกัน คือช่วยหารบิล/สรุปว่าใครต้องจ่ายใครเท่าไร  
แต่ถ้ามองจาก “ระดับผลิตภัณฑ์” จะต่างกันชัด:

- **Harn-Gan** เด่นเรื่องความยืดหยุ่น ความโปร่งใส และการคำนวณละเอียด
- **KBank Kuntong** เด่นเรื่องความสะดวก การอยู่ใน ecosystem ที่มีความน่าเชื่อถือ และการใช้งานกับเงินจริง

พูดง่าย ๆ:
- Harn-Gan เหมาะกับคนที่อยาก “คุมกติกาเอง”
- Kuntong เหมาะกับคนที่อยาก “ใช้ง่ายและจบงานเร็ว”

---

## Harn-Gan — ข้อดี

### 1) ยืดหยุ่นมาก
จากโค้ด Harn-Gan รองรับเคสที่หลายแอปทำไม่ครบ เช่น:
- แบ่งแบบ item-based
- แบ่งแบบเท่า ๆ กัน
- ค่าบริการ / VAT / ค่าส่ง / ส่วนลด
- คนฟรี
- คนมาทีหลัง
- คนเดียวออกก่อน
- หลายคนช่วยกันจ่าย

นี่คือจุดแข็งหลักสุด เพราะมันไม่บังคับ user ให้อยู่ในรูปแบบเดียว

### 2) โปร่งใส ตรวจสอบได้
ตรรกะคำนวณอยู่ใน `utilities/calculations.js` ทำให้:
- ดูได้ว่าคิดยังไง
- trace ปัญหาได้ง่าย
- แก้ rule ได้ตรงจุด
- เหมาะกับการพัฒนา/ทดลอง logic ใหม่

ถ้า user หรือ developer อยากรู้ว่า “ทำไมยอดนี้ออกมาแบบนี้” มันอธิบายได้ชัดกว่าระบบที่ซ่อน logic ไว้ในแพลตฟอร์ม

### 3) คิด settlement ชัดเจน
Step 4 และหน้าผลลัพธ์ของ Harn-Gan พยายามทำให้เห็นชัดว่า:
- ใครออกก่อน
- ออกไปก่อนเท่าไร
- คนอื่นต้องคืนเท่าไร
- สุดท้ายคนออกก่อนเหลือสุทธิเท่าไร

อันนี้ดีมากสำหรับลดความสับสนในวงเพื่อน

### 4) แยก logic ออกจาก UI ค่อนข้างดี
ฝั่ง React เป็นตัวรับ input และแสดงผล  
ฝั่ง backend/calc เป็นตัวคำนวณ  
ทำให้:
- test ง่ายขึ้น
- refactor UI ได้โดยไม่พัง logic
- คุมความถูกต้องของยอดได้ดีขึ้น

### 5) เหมาะกับเคสจริงแบบจุกจิก
เช่น:
- มีคนไม่กินบางรายการ
- มีคนมาช้า
- มีคนฟรี
- มีหลาย payer
- มีส่วนลด/ค่าบริการ/ค่าส่ง

นี่คือจุดที่ Harn-Gan “เก่ง” กว่า split bill ทั่วไป

## Harn-Gan — ข้อด้อย

### 1) ยังไม่ใช่ระบบโอนเงินจริง
นี่คือข้อด้อยใหญ่สุดเมื่อเทียบกับ Kuntong:
- ไม่โอนเงิน
- ไม่ request transfer
- ไม่ track ว่าโอนจริงหรือยัง
- ไม่เชื่อมบัญชี/วอลเล็ต

สรุปคือมันเก่งเรื่องคำนวณ แต่ไม่เก่งเรื่อง “ลงมือจ่ายจริง”

### 2) ต้องกรอกข้อมูลเยอะ
เพราะรองรับ rule เยอะ ผู้ใช้ต้องใส่รายละเอียดมาก:
- เพื่อน
- รายการอาหาร
- คนกินแต่ละรายการ
- ค่าบริการ
- VAT
- ค่าส่ง
- ส่วนลด
- คนฟรี
- วิธีจ่าย

สำหรับ power user ดีมาก แต่สำหรับ user ทั่วไปอาจรู้สึกหนัก

### 3) UX มี validation เยอะ
เพราะกติกาเข้ม:
- step 2 ต้องกรอกครบก่อน
- free people ห้ามเป็น payer
- multi payer ต้องยอดตรงเป๊ะ
- payer ต้องอยู่ในรายชื่อ

ถูกทางด้าน logic แต่เพิ่ม friction ให้ผู้ใช้

### 4) ขาด trust layer แบบแพลตฟอร์มการเงิน
Harn-Gan เป็นแอป standalone  
ไม่ได้มี:
- login แบบ bank ecosystem
- identity layer
- social graph หรือ group context ที่ผู้ใช้คุ้นเคย
- ความรู้สึกว่าเป็นระบบโอนเงินจริงที่ “มั่นใจได้”

## KBank Kuntong — ข้อดี

> หมายเหตุ: ส่วนนี้เป็นการประเมินในเชิง product-level ของระบบแบบ Kuntong/บริการหารบิลใน ecosystem ธนาคาร ไม่ใช่การอ่าน source code ของ Kuntong โดยตรง

### 1) เชื่อมกับ ecosystem การเงินจริง
ข้อได้เปรียบใหญ่สุดของ Kuntong คืออยู่ในระบบที่เกี่ยวกับเงินจริง:
- โอนง่าย
- จัดการ settlement ได้สะดวก
- ผู้ใช้เชื่อถือได้มากกว่า
- เหมาะกับการ “จ่ายจริง” มากกว่าการแค่คำนวณ

### 2) ใช้งานง่ายกับผู้ใช้ทั่วไป
คนส่วนใหญ่คุ้นกับบัญชีธนาคาร/แอปที่ใช้ประจำอยู่แล้ว  
ดังนั้น flow แบบนี้มัก:
- เข้าใจง่าย
- ขั้นตอนน้อยกว่า
- friction ต่ำกว่า
- เหมาะกับ user mass market

### 3) มี trust สูงกว่า
เมื่ออยู่ใน ecosystem ธนาคาร:
- ผู้ใช้รู้สึกปลอดภัยกว่า
- เชื่อมกับ transaction จริง
- ลดความลังเลเวลาจะจ่ายเงิน

### 4) เหมาะกับการใช้งานจริงในชีวิตประจำวัน
ถ้าจุดประสงค์คือ “หารบิลแล้วให้จบด้วยการจ่ายเงินจริง”  
Kuntong จะมีความเป็น product ที่พร้อมใช้งานมากกว่า

---

## KBank Kuntong — ข้อด้อย

### 1) ไม่โปร่งใสเท่า Harn-Gan
ระบบใน ecosystem ใหญ่ ๆ มักเน้น simplicity มากกว่าการเปิด rule ทุกอย่าง  
ผลคือ:
- ผู้ใช้อาจไม่เห็นว่าคำนวณยังไง
- ปรับ rule ละเอียดได้ยาก
- debug หรือ audit ยากกว่า

### 2) ความยืดหยุ่นน้อยกว่า
Kuntong หรือ product แบบ bank-native มักถูกออกแบบให้ standard:
- แบ่งบิลแบบซับซ้อนอาจไม่สะดวก
- item-based deep split อาจไม่ได้ละเอียดเท่า Harn-Gan
- special cases เช่น free people / late joiners อาจไม่ใช่จุดเด่น

### 3) พึ่งพา ecosystem
ถ้าอยู่ในระบบธนาคาร/LINE:
- portable น้อยกว่า
- ขึ้นกับ rule ของ platform
- developer ปรับ logic เองได้น้อยกว่า
- ทำเป็น sandbox ทดลองยากกว่า

### 4) ไม่เหมาะถ้าอยาก “คุมกติกาเอง”
ถ้าผู้ใช้ต้องการตั้ง rule พิเศษเยอะ ๆ  
ระบบแบบ Kuntong มักตอบโจทย์น้อยกว่า Harn-Gan

## คำตอบแบบสั้นที่สุด
- **Harn-Gan**: เก่งเรื่อง “คำนวณและอธิบายการหารบิล”
- **Kuntong**: เก่งเรื่อง “เชื่อมการหารบิลเข้ากับการจ่ายเงินจริง”

ดังนั้นสองตัวนี้ไม่ได้แข่งกันตรง ๆ แบบตัวไหนดีกว่าทุกด้าน  
แต่เป็นคนละชั้นของปัญหา:

- Harn-Gan แก้ **logic / math / rule modeling**
- Kuntong แก้ **payment workflow / ecosystem / convenience**

ถ้าคุณต้องการ ผมสามารถต่อให้เป็น **ตารางสรุปแบบ product strategy** หรือ **วิเคราะห์ว่า Harn-Gan ควรเพิ่มอะไรเพื่อเข้าใกล้ Kuntong** ได้อีกครับ
